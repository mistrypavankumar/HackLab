# HackLab Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a localhost-only Next.js app where developers learn web security by exploiting deliberately vulnerable labs and watching the fix defuse the same attack.

**Architecture:** Next.js 16 App Router + Tailwind. Each lab is a self-contained folder registered in a central registry. API routes execute a `vulnerable` or `secure` code path against a seeded local SQLite DB. A localhost-only middleware hard-gates the app.

**Tech Stack:** Next.js 16, React 19, TypeScript (strict), Tailwind CSS, better-sqlite3, jsonwebtoken, Shiki, Vitest.

---

## File Structure

```
hacklab/
  package.json, tsconfig.json, next.config.ts, tailwind config, postcss
  middleware.ts                      # localhost-only hard gate
  src/
    app/
      layout.tsx, globals.css        # shell + safety banner
      page.tsx                       # home: categories + lab grid
      labs/[slug]/page.tsx           # generic lab page renderer
      api/labs/<slug>/.../route.ts   # per-lab vulnerable/secure endpoints
    components/
      SafetyBanner.tsx, LabLayout.tsx, ModeToggle.tsx, CodeBlock.tsx, LabCard.tsx
    labs/
      types.ts                       # Lab, Category, Difficulty types
      registry.ts                    # ordered list of all lab metas
      <slug>/meta.ts                 # metadata + code snippets
      <slug>/Lab.tsx                 # interactive client component
    lib/
      db.ts                          # better-sqlite3 connection
      seed.ts                        # idempotent seed
      progress.ts                    # localStorage completed marks
      sanitize.ts, rateLimit.ts, jwt.ts  # shared secure helpers
  tests/                             # Vitest secure-path tests
```

## Lab task template (applies to Tasks 6–13)

Each lab task:
1. **Test first:** `tests/<slug>.test.ts` asserts the **secure** path blocks the attack and the **vulnerable** path allows it.
2. Run → fail.
3. Implement API route(s) with both `vulnerable` and `secure` branches + `meta.ts` (with real code snippets) + `Lab.tsx` interactive panel + append to `registry.ts`.
4. Run test → pass; manual exploit check in browser.
5. Commit `feat: <slug> lab`.

### Task 6: Stored XSS (`frontend`)
- API: `GET /api/labs/stored-xss` (list comments), `POST` (add). Vulnerable: stores+returns raw body; client renders with `dangerouslySetInnerHTML`. Secure: sanitize via `lib/sanitize.ts` (escape `<>&"'` / strip tags) and render as text.
- Lab.tsx: comment form + feed; payload hint `<img src=x onerror=alert(1)>`. Toggle switches `mode`.
- Test: `sanitize('<img onerror=x>')` contains no `onerror`/`<img`; vulnerable store returns raw.

### Task 7: CSRF (`frontend`)
- API: `POST /api/labs/csrf/change-email`. Vulnerable: accepts request with only a session cookie. Secure: requires matching `csrfToken` (double-submit) + checks `SameSite`-style token.
- Lab.tsx: shows current email, a legit form, and a simulated cross-site form (no token) — succeeds in vulnerable, rejected (403) in secure.
- Test: secure handler rejects missing/mismatched token (403); accepts valid token.

### Task 8: SQL Injection (`backend`)
- API: `POST /api/labs/sql-injection/login` and `/search`. Vulnerable: string-concatenated SQL (`db.prepare("... WHERE username='"+u+"'")` via raw exec). Secure: parameterized `?` bindings.
- Lab.tsx: login form; payload `' OR '1'='1' --` logs in as admin in vulnerable; fails in secure. Search shows data exfil.
- Test: vulnerable login with `' OR '1'='1` returns a user; secure login returns null.

### Task 9: IDOR (`backend`)
- API: `GET /api/labs/idor/invoice?id=&as=`. Vulnerable: returns invoice by id with no ownership check. Secure: returns 403 unless `invoice.user_id === currentUserId`.
- Lab.tsx: logged in as user #2; fetch own invoice, then change id to #1 — leaks in vulnerable, 403 in secure.
- Test: secure returns 403 for non-owned id; vulnerable returns the row.

### Task 10: JWT tampering (`auth`)
- `lib/jwt.ts`: sign/verify helpers. API: `POST /verify`. Vulnerable: decode without verifying signature / accepts `alg:none`. Secure: `jwt.verify` with HS256 + strong secret, `algorithms:['HS256']`.
- Lab.tsx: issue a user token, edit payload to `role:admin` (alg=none) — accepted in vulnerable, rejected in secure.
- Test: secure verify throws on alg=none and on wrong-secret token; accepts valid.

### Task 11: Brute force (`auth`)
- `lib/rateLimit.ts`: in-memory attempt counter keyed by ip+user. API: `POST /api/labs/brute-force/login`. Vulnerable: unlimited attempts. Secure: lock after 5 failures / window.
- Lab.tsx: an "auto-attack" button firing N attempts; vulnerable keeps trying, secure returns 429 after threshold.
- Test: secure handler returns locked=true after 5 failed attempts; vulnerable never locks.

### Task 12: Exposed secrets (`misconfig`)
- API: `GET /api/labs/exposed-secrets/config`. Vulnerable: returns `process.env` subset incl. fake `API_SECRET`, `DB_PASSWORD`. Secure: endpoint returns 404 / only safe public flags.
- Lab.tsx: fetch config; vulnerable leaks secrets, secure returns nothing sensitive.
- Test: secure response contains no key matching `/secret|password|key/i`; vulnerable does.

### Task 13: Security headers / clickjacking (`misconfig`)
- API: `GET /api/labs/security-headers/page`. Vulnerable: no CSP/X-Frame-Options. Secure: sets `Content-Security-Policy: frame-ancestors 'none'` + `X-Frame-Options: DENY`.
- Lab.tsx: attempts to load the endpoint in an `<iframe>` — frames in vulnerable, blocked in secure; shows header diff.
- Test: secure response headers include CSP `frame-ancestors 'none'` and `X-Frame-Options: DENY`; vulnerable omits them.
